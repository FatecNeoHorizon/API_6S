# Non-Relational Database Documentation

## Overview

This MongoDB schema models physical infrastructure and operational metrics of Brazil's energy distribution networks. The design supports two primary objectives:

1. **Infrastructure Dimensioning:** Identify network capacity and geographic distribution patterns to assess market potential for Tecsys technology (sensing and telemetry).
2. **Loss Prediction:** Geographically locate energy losses and continuity failures to guide infrastructure improvements and predict where Tecsys deployment would have the highest impact.

The data-driven approach enables regulatory compliance (ANEEL reporting) while creating actionable insights for reducing non-technical losses (theft, fraud) and improving service continuity.

## Technology Stack

**Database:** `tecsys` (MongoDB 6.0)  
**Driver:** PyMongo  
**Connection:** Configured via environment variables using `python-dotenv`

## Setup Overview

`database.py` defines all collections, their JSON Schema validators, and indexes in a single idempotent `setup()` function called at application startup. This ensures the database schema is automatically initialized on first run and remains immutable through subsequent executions.

**Input files (uploaded by the user):**

| File                                                                | Type  | Produces                                           |
| ------------------------------------------------------------------- | ----- | -------------------------------------------------- |
| `Base_de_Dados_das_Perdas_de_Energia_nos_Processos_Tarifários.xlsx` | Excel | `energy_losses_tariff`                             |
| `EDP_SP_<distributor>_<date>.gdb.zip`                               | GDB   | `substations`, `distribution_transformers`, `conj` |
| `indicadores-continuidade-coletivos-{start_year}-{end_year}.csv`    | CSV   | `conj` (embedded indicators)                       |
| `indicadores-continuidade-coletivos-limite.csv`                     | CSV   | `conj` (embedded limits)                           |

**Recommended load order:**

1. `energy_losses_tariff` — annual regulatory data per distributor
2. `substations` — substation polygons from GDB
3. `distribution_transformers` — transformer points from GDB
4. `conj` — UC set polygons (GDB layer `CONJ`) + embedded indicators (CSV) + embedded limits (CSV)
5. `distribution_indices` — operational records, loaded in chunks

---

## Functions

### `get_client()`

Builds the MongoDB connection URI from environment variables and returns a `MongoClient` instance. Authentication credentials are securely loaded from environment variables (never hardcoded or exposed in documentation).

### `setup()`

Creates all collections and indexes if they do not already exist. Uses `db.list_collection_names()` to check existence before creation — making the function fully **idempotent** (safe to run multiple times without duplicating data).

### 4. `conj`

**Sources (embedded — 3 sources merged into 1 collection):**

1. GDB layer `CONJ` — geographic boundaries and descriptive metadata per UC set
2. `indicadores-continuidade-coletivos-{start_year}-{end_year}.csv` — DEC/FEC measured values per set per period
3. `indicadores-continuidade-coletivos-limite.csv` — regulatory DEC/FEC limits per set

**Granularity:** One document per UC set  
**Geometry:** MultiPolygon  
**Validation:** `strict`

Central analytical collection linking continuity metrics to geographic infrastructure. Stores the UC set polygon (GDB), its measured DEC/FEC values (CSV indicators), and the regulatory limits for those values (CSV limits) in a single embedded document — avoiding `$lookup` joins on every map or TAM/SAM query.

> **Embedded rationale:** The three sources are always read together (geometry for map rendering, indicators for current values, limits for violation detection). Embedding eliminates runtime joins against the 5M-row indicators CSV. The trade-off is write complexity: updating an indicator requires modifying the embedded array inside the set document, rather than updating a standalone row.

| `shape_length`         | double                          | ❌       | Perimeter of the UC set boundary in map units. Mapped from `Shape_Length`.                                     |

## Data Source Cross-Reference

This section documents the relationships that only become visible when Alexandre's analysis (energy losses and physical infrastructure) and Gabriel's analysis (continuity indicators and UC sets) are examined together.

### Overview of Each Analysis

| Analysis                                     | Pattern                                 | Sources                                        | Output collections                                                 |
| -------------------------------------------- | --------------------------------------- | ---------------------------------------------- | ------------------------------------------------------------------ |
| Physical infrastructure + losses (Alexandre) | 1 source → 1 collection                 | GDB (`SUB`, `UN_TRA_D`), Excel (perdas)        | `substations`, `distribution_transformers`, `energy_losses_tariff` |
| Continuity indicators + UC sets (Gabriel)    | **Embedded** — 3 sources → 1 collection | GDB (`CONJ`), CSV (indicadores), CSV (limites) | `conj`                                                             |

### `conj` as the Central Link

`conj` is the entity that bridges both analyses. It holds the geographic boundaries of each UC set (Gabriel's GDB data) and the DEC/FEC values and limits (Gabriel's CSVs) — but its `distributor_code` is the same key used in Alexandre's infrastructure collections (`substations`, `distribution_transformers`), making it the join point between the two analyses.

```
                         ┌───────────────────────────────────┐
                         │                conj               │
                         │  GDB CONJ + distribution_indices  │
                         └───────┬──────────────┬────────────┘
                                 │              │
              ┌──────────────────┘              └──────────────────────┐
     code ────▼────                                      ──────────────▼──── distributor_code
              │                                                        │
   ┌──────────▼───────────────┐               ┌────────────────────────▼──────────┐
   │  distribution_indices    │               │  substations                      │
   │  consumer_unit_set_id    │               │  distribution_transformers        │
   └──────────────────────────┘               │  energy_losses_tariff             │
                                              └───────────────────────────────────┘
```

### Link Key Consistency Notes

| Issue                        | Detail                                                                                                     | Resolution                                           |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| CONJ join key type           | `IdeConjUndConsumidoras` is `int64` in CSV; `conj.code` must be `string`                                   | Both sides cast to `string` at ETL                   |
| `VlrIndiceEnviado` format    | Values use comma decimal separator (`,44` = `0.44`)                                                        | ETL replaces `,` with `.` before parsing to `double` |
| `NumCNPJ` type               | Arrives as `int64` in CSV                                                                                  | Cast to `string` at ETL to preserve formatting       |
| No canonical distributor key | `agent_acronym` (CSV) and `distributor_code` (GDB) reference the same entity but cannot be directly joined | Joined via `conj`, which holds both fields           |

## Cross-Collection Relationships

```
conj.code              ←→  distribution_indices.consumer_unit_set_id
conj.distributor_code  ←→  substations.distributor_code
conj.distributor_code  ←→  distribution_transformers.distributor_code

distribution_indices.agent_acronym  ←→  energy_losses_tariff.distributor  ⚠️ no canonical key

conj.geometry  $geoIntersects  distribution_transformers.geometry
```

> **Legend:**  
> `←→` = join key (both sides)  
> `⚠️` = no canonical key — `agent_acronym` (CSV) and distributor name (Excel) require ETL-level mapping  
> `$geoIntersects` = spatial join resolved at query time, no stored foreign key
